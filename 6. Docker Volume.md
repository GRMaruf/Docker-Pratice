#  Docker Volumes

## The Problem

Imagine you have this running container:

```text
Container
│
├── Django
├── Python
├── SQLite Database
├── Uploaded Images
└── Your Project
```

Now you remove the container:

```bash
docker rm django-app
```

Everything **inside the container** is gone.

> **Containers are ephemeral.** They're designed to be created and destroyed.

---

## But wait...

Suppose a user registers on your website.

The data is saved to:

```text
db.sqlite3
```

If `db.sqlite3` lives only inside the container:

```text
Container
└── db.sqlite3
```

and you remove the container:

```bash
docker rm django-app
```

Then:

```text
❌ Database gone
❌ Uploaded images gone
❌ Generated files gone
```

That's obviously unacceptable for real applications.

---

# Solution: Volumes

A **volume** is storage managed by Docker that exists **outside** the container.

```text
Container
     │
     │
     ▼
Docker Volume
```

Now when the container is removed:

```text
Container ❌

Volume ✅
```

The data survives.

---

# Analogy

Imagine you're staying in a hotel.

The hotel room is your **container**.

Your luggage is the **volume**.

If you check out:

```text
Hotel Room ❌
```

Your luggage still exists because it wasn't part of the room.

---

# Create a Volume

```bash
docker volume create django-data
```

See all volumes:

```bash
docker volume ls
```

Example:

```text
DRIVER    VOLUME NAME
local     django-data
```

Inspect it:

```bash
docker volume inspect django-data
```

---

# Mount a Volume

Suppose you want to persist the SQLite database.

You could mount a volume:

```bash
docker run \
-v django-data:/app/data \
mysite
```

This means:

```text
Docker Volume
        │
        ▼
/app/data
```

Anything written to `/app/data` is stored in the volume.

---

# Bind Mounts vs Named Volumes

These are often confused, so let's separate them.

## 1. Named Volume

```bash
-v django-data:/app/data
```

Docker manages the storage location.

You don't care where it lives on disk.

```
Docker
└── django-data
```

This is great for:

* PostgreSQL
* MySQL
* Redis
* Production databases

---

## 2. Bind Mount

```bash
-v $(pwd):/app
```

This maps your current project directory directly into the container.

```text
Your Laptop
│
├── manage.py
├── core/
└── myApp/

↓

Container

/app
```

Now, if you edit:

```python
views.py
```

the change is immediately visible inside the container.

No rebuild required.

---

# Why Developers Love Bind Mounts

Without a bind mount:

```text
Edit code

↓

docker build

↓

docker run

↓

Repeat
```

This gets slow.

With a bind mount:

```text
Edit code

↓

Refresh browser

↓

Done ✅
```

Much faster.

---

# Example for Django Development

```bash
docker run \
-p 8000:8000 \
-v $(pwd):/app \
mysite
```

Now:

1. Open `views.py`.
2. Make a change.
3. Save it.
4. Refresh your browser.

The container sees the updated file because it's using your local project directory.

> **Note:** Depending on your shell or platform, `$(pwd)` may differ. In GitHub Codespaces (Linux), `$(pwd)` works as shown.

# Volumes vs Images

An important distinction:

### Image

Contains:

* Python
* Django
* Installed packages
* Your application code (at build time)

It **should not** contain changing data like databases or user uploads.

---

### Volume

Contains:

* Databases
* Uploaded media
* User-generated files
* Anything that needs to survive container replacement

---

# Useful Volume Commands

Create:

```bash
docker volume create mydata
```

List:

```bash
docker volume ls
```

Inspect:

```bash
docker volume inspect mydata
```

Remove:

```bash
docker volume rm mydata
```

Remove unused volumes:

```bash
docker volume prune
```

---

# Development vs Production

### Development

A bind mount is usually the best choice:

```bash
-v $(pwd):/app
```

because you're constantly editing code.

### Production

You typically:

* build the application into the image,
* use **named volumes** only for persistent data such as databases and uploaded media.
