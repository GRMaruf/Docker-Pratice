# Django + PostgreSQL with Docker Compose

Let's move from this:

```text
Django
   │
SQLite
```

to this:

```text
             Docker Compose
                  │
        ┌─────────┴─────────┐
        │                   │
   Django Container   PostgreSQL Container
        │                   │
        └─────────┬─────────┘
                  │
            Docker Network
```

Notice that **PostgreSQL is just another container**.

---

# Create `docker-compose.yml`

Replace your current file with:

```yaml
services:
  web:
    build: .
    command: python manage.py runserver 0.0.0.0:8000

    volumes:
      - .:/app

    ports:
      - "8000:8000"

    depends_on:
      - db

    environment:
      DB_NAME: mydatabase
      DB_USER: myuser
      DB_PASSWORD: mypassword
      DB_HOST: db
      DB_PORT: 5432

  db:
    image: postgres:17

    environment:
      POSTGRES_DB: mydatabase
      POSTGRES_USER: myuser
      POSTGRES_PASSWORD: mypassword

    volumes:
      - postgres_data:/var/lib/postgresql/data

volumes:
  postgres_data:
```

---

# Where did the database come from?

This line:

```yaml
image: postgres:17
```

tells Docker:

> Download the official PostgreSQL image.

When the container starts for the **first time**, PostgreSQL reads these variables:

```yaml
POSTGRES_DB: mydatabase
POSTGRES_USER: myuser
POSTGRES_PASSWORD: mypassword
```

and automatically creates:

* Database: `mydatabase`
* User: `myuser`
* Password: `mypassword`

You do **not** need to run:

```sql
CREATE DATABASE ...
```

or

```sql
CREATE USER ...
```

The official PostgreSQL image performs that initialization for you on its first startup.

---

# When does this happen?

Only when the PostgreSQL data directory is empty.

Think of it like this:

```
First run

↓

Volume is empty

↓

Create database

↓

Create user

↓

Save data in volume
```

On later runs:

```
Volume already contains data

↓

Don't recreate anything
```

That's why the named volume is important.

# Connect Django to PostgreSQL

Install the PostgreSQL driver:

```bash
pip install psycopg[binary]
```

Then update your `requirements.txt`:

```bash
pip freeze > requirements.txt
```

---

# Update `settings.py`

Instead of SQLite:

```python
DATABASES = {
    "default": {
        "ENGINE": "django.db.backends.sqlite3",
        "NAME": BASE_DIR / "db.sqlite3",
    }
}
```

use:

```python
import os

DATABASES = {
    "default": {
        "ENGINE": "django.db.backends.postgresql",
        "NAME": os.getenv("DB_NAME"),
        "USER": os.getenv("DB_USER"),
        "PASSWORD": os.getenv("DB_PASSWORD"),
        "HOST": os.getenv("DB_HOST"),
        "PORT": os.getenv("DB_PORT"),
    }
}
```

Now Django reads the connection details from the environment variables provided by Docker Compose.

---

# Build and Start

```bash
docker compose up --build
```

Docker will:

1. Build your Django image.
2. Download the PostgreSQL image (if needed).
3. Create the `postgres_data` volume.
4. Start PostgreSQL.
5. Create the database and user (first run only).
6. Start Django.

---

# Run Migrations

Open a shell in the Django container:

```bash
docker compose exec web bash
```

Run:

```bash
python manage.py migrate
```

Then create an admin user:

```bash
python manage.py createsuperuser
```

Exit:

```bash
exit
```

---

# Verify the Database

You can also enter the PostgreSQL container:

```bash
docker compose exec db bash
```

Start the PostgreSQL client:

```bash
psql -U myuser -d mydatabase
```

Useful commands inside `psql`:

```sql
\l     -- list databases
\du    -- list users (roles)
\dt    -- list tables
```

Exit:

```sql
\q
```

## A small improvement we'll make next

Right now, our Compose file contains:

```yaml
POSTGRES_PASSWORD: mypassword
```

Hardcoding credentials is fine for learning, but it's not a good practice for real projects.

In the next step, we'll introduce a `.env` file and configure both Docker Compose and Django to use it, so your passwords and other secrets stay out of your configuration files. That's the pattern you'll see in most professional Dockerized Django applications.
