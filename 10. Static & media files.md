# Case 1: Static Files

```nginx
location /static/ {
    alias /app/staticfiles/;
}
```

Suppose the browser requests:

```
GET /static/css/style.css
```

Nginx checks its configuration.

It sees:

```nginx
location /static/
```

So it **doesn't contact Django at all**.

Instead, it looks directly on the filesystem:

```
/app/staticfiles/css/style.css
```

If the file exists:

```
Browser

↓

Nginx

↓

Filesystem

↓

style.css

↓

Browser
```

Django and Gunicorn are never involved.

This is extremely fast because Nginx is optimized for serving files.

---

# Case 2: Dynamic Requests

```nginx
location / {
    proxy_pass http://web:8000;
}
```

Now the browser requests:

```
GET /
```

or

```
GET /products/
```

or

```
GET /api/users/
```

Nginx matches:

```nginx
location /
```

Since there is no static file to serve, it forwards the request to Gunicorn:

```
Browser

↓

Nginx

↓

Gunicorn (web:8000)

↓

Django URL Dispatcher

↓

View

↓

Database

↓

Response

↓

Nginx

↓

Browser
```

---

# A Complete Example

Suppose your site has:

```
/
```

```
/about/
```

```
/products/
```

```
/static/css/style.css
```

```
/static/js/app.js
```

Nginx routes them like this:

| URL                     | Who Handles It?   |
| ----------------------- | ----------------- |
| `/`                     | Gunicorn → Django |
| `/about/`               | Gunicorn → Django |
| `/products/`            | Gunicorn → Django |
| `/static/css/style.css` | Nginx             |
| `/static/js/app.js`     | Nginx             |

---

# Why Not Send Everything to Django?

Imagine a page loads:

* 1 HTML page
* 3 CSS files
* 8 JavaScript files
* 20 images

That's **32 HTTP requests**.

If every request went through Django:

```
32 requests

↓

Gunicorn

↓

Django
```

Django would waste time serving files that never change.

Instead:

```
31 static files

↓

Nginx

↓

Filesystem
```

```
1 dynamic request

↓

Gunicorn

↓

Django
```

This is much more efficient.

# How Does Nginx Choose?

Suppose your configuration is:

```nginx
server {

    location /static/ {
        alias /app/staticfiles/;
    }

    location / {
        proxy_pass http://web:8000;
    }

}
```

### Request 1

```
GET /static/logo.png
```

Matches:

```nginx
location /static/
```

Result:

```
Serve the file directly.
```

---

### Request 2

```
GET /accounts/login/
```

Doesn't match `/static/`, so it falls back to:

```nginx
location /
```

Result:

```
Forward to Django.
```

---

### Request 3

```
GET /api/books/
```

Again:

```
location /
```

↓

```
Gunicorn

↓

Django
```

# Why Use `alias` Instead of `proxy_pass`?

These directives have different purposes:

* **`alias`** tells Nginx: "The response is already a file on disk. Read it and send it."
* **`proxy_pass`** tells Nginx: "I don't generate this response. Forward the request to another server and return whatever it sends back."

So:

```nginx
location /static/ {
    alias /app/staticfiles/;
}
```

means:

> "Look in this directory on disk."

Whereas:

```nginx
location / {
    proxy_pass http://web:8000;
}
```

means:

> "Ask the Django application running in the `web` container."

## A mental model

Think of Nginx as a receptionist:

```
Customer asks for:

"style.css"

↓

Receptionist walks to the filing cabinet

↓

Returns the file
```

No need to bother the developer.

But if the customer asks:

```
"Show me all products under $100"

↓

Receptionist forwards the request

↓

Django application processes it

↓

Database is queried

↓

Response comes back
```

The key idea is:

* **Static content** (CSS, JavaScript, images, fonts) is served directly from disk by Nginx using `alias` (or sometimes `root`).
* **Dynamic content** (HTML pages, APIs, forms, authentication, database queries) is forwarded to Gunicorn with `proxy_pass`, and Django generates the response.
