# Step 3: Working with Containers

Up to now, you've built an **image** and run a **container**.

Let's expand the picture:

```text
Your Django Code
       │
       ▼
Dockerfile
       │
       ▼
docker build
       │
       ▼
Docker Image
       │
       ▼
docker run
       │
       ▼
Running Container
```

The **image** is static. The **container** is a live, running environment.

---

# 1. Give Your Container a Name

Instead of letting Docker assign a random name, you can choose one:

```bash
docker run --name django-app -p 8000:8000 mysite
```

Now you can refer to it by name instead of by its ID.

Check your running containers:

```bash
docker ps
```

Example output:

```text
CONTAINER ID   IMAGE    STATUS          PORTS                  NAMES
4b3c8d...      mysite   Up 2 minutes    0.0.0.0:8000->8000     django-app
```

---

# 2. See All Containers

Only running containers:

```bash
docker ps
```

Running + stopped containers:

```bash
docker ps -a
```

You might see:

```text
CONTAINER ID   IMAGE
abcd1234       mysite
efgh5678       mysite
ijkl9012       mysite
```

These are different containers created from the same image.

---

# 3. Stop a Container

By ID:

```bash
docker stop abcd1234
```

Or by name:

```bash
docker stop django-app
```

---

# 4. Start It Again

Stopping a container doesn't delete it.

Restart it with:

```bash
docker start django-app
```

---

# 5. Remove a Container

If you no longer need it:

```bash
docker rm django-app
```

If it's still running, stop it first:

```bash
docker stop django-app
docker rm django-app
```

---

# 6. Detached Mode (`-d`)

So far, you've probably run:

```bash
docker run -p 8000:8000 mysite
```

The terminal stays attached to the container, showing Django's output.

To run it in the background:

```bash
docker run -d --name django-app -p 8000:8000 mysite
```

Now your prompt returns immediately.

Check that it's running:

```bash
docker ps
```

---

# 7. View Logs

When a container runs in the background, you won't see its output automatically.

Use:

```bash
docker logs django-app
```

Or follow the logs in real time:

```bash
docker logs -f django-app
```

This is similar to watching the Django development server's console output.

---

# 8. Enter a Running Container

This is one of the most useful commands.

Open a shell inside the running container:

```bash
docker exec -it django-app bash
```

If `bash` isn't available:

```bash
docker exec -it django-app sh
```

Now you're inside the container:

```text
root@container:/app#
```

You can run commands like:

```bash
python --version
ls
pwd
python manage.py check
python manage.py migrate
```

You're interacting with the environment inside the container, not your Codespace directly.

Exit with:

```bash
exit
```

---

# 9. Interactive vs Detached Mode

### Interactive

```bash
docker run mysite
```

* Terminal is attached.
* Pressing `Ctrl + C` stops the container.

### Detached

```bash
docker run -d mysite
```

* Runs in the background.
* Use `docker logs` to see output.
* Stop it with `docker stop`.

---

# 10. Create a `.dockerignore`

Just as Git uses `.gitignore`, Docker uses `.dockerignore`.

Create a file named:

```text
.dockerignore
```

For your Django project, start with:

```text
.git
.gitignore
__pycache__/
*.pyc
*.pyo
*.pyd
.env
venv/
.env/
db.sqlite3
media/
README.md
```

### Why?

When you run:

```bash
docker build .
```

Docker sends the **build context** (your project files) to the Docker daemon.

Without a `.dockerignore`, it sends everything:

* `.git`
* virtual environments
* caches
* local databases
* temporary files

Ignoring unnecessary files:

* speeds up builds,
* reduces image size,
* avoids accidentally including secrets like `.env`.
