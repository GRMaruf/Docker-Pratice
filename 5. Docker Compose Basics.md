# Docker Compose

Imagine your Django application grows.

Instead of just Django, you now need:

* Django
* PostgreSQL
* Redis
* Nginx

Without Docker Compose, you would need to run four different `docker run` commands and manually connect them.

```text
docker run ...
docker run ...
docker run ...
docker run ...
```

That's difficult to maintain.

---

# What is Docker Compose?

Docker Compose lets you describe your entire application in one YAML file.

Think of it as:

> **"This is my application. These are all the containers it needs, and this is how they connect."**

Instead of typing long commands every time, you just write:

```bash
docker compose up
```

Docker creates everything for you.

---

# Our Goal

We'll eventually have something like this:

```text
               Docker Compose
                     │
        ┌────────────┴────────────┐
        │                         │
   Django Container         PostgreSQL Container
        │                         │
        └────────────┬────────────┘
                     │
                 Docker Network
```

Notice something interesting:

**We never manually connect them.**

Docker Compose automatically creates a private network.

# Your First `docker-compose.yml`

Create a file named:

```text
docker-compose.yml
```

Add this:

```yaml
services:
  web:
    build: .
    ports:
      - "8000:8000"
```

Let's understand every line.

---

## `services`

```yaml
services:
```

This is where you define the containers your application needs.

Today:

* Django

Later:

* PostgreSQL
* Redis
* Nginx
* Celery
* Flower

---

## `web`

```yaml
web:
```

This is just a service name.

It could be:

```yaml
django:
backend:
app:
website:
```

Most Django projects simply call it `web`.

---

## `build`

```yaml
build: .
```

This tells Docker Compose:

> Build an image using the `Dockerfile` in the current directory.

It's equivalent to:

```bash
docker build .
```

---

## `ports`

```yaml
ports:
  - "8000:8000"
```

Meaning:

```text
Codespace Port 8000
        │
        ▼
Container Port 8000
```

---

# Start Everything

Instead of:

```bash
docker run -p 8000:8000 mysite
```

you now run:

```bash
docker compose up
```

Docker Compose will:

1. Build the image (if needed)
2. Create the container
3. Start the container
4. Show the logs

---

# Detached Mode

Run it in the background:

```bash
docker compose up -d
```

Exactly like:

```bash
docker run -d ...
```

---

# Stop Everything

```bash
docker compose down
```

This stops and removes the containers created by Compose.

---

# Rebuild After Changes

If you change the `Dockerfile` or install new packages:

```bash
docker compose up --build
```

This rebuilds the image before starting the containers.

---

# Inspect the Services

Running containers:

```bash
docker compose ps
```

Logs:

```bash
docker compose logs
```

Live logs:

```bash
docker compose logs -f
```

# Adding PostgreSQL

Let's extend the file:

```yaml
services:
  web:
    build: .
    ports:
      - "8000:8000"

  db:
    image: postgres:17
    environment:
      POSTGRES_DB: mydb
      POSTGRES_USER: myuser
      POSTGRES_PASSWORD: mypassword
```

Now Compose will create **two containers**:

```text
web
db
```

---

# Automatic Networking

This is one of Docker Compose's best features.

Inside the Django container, you **do not** connect to PostgreSQL using:

```text
localhost
```

Instead, use the service name:

```python
HOST = "db"
```

Why?

Because Compose automatically creates DNS entries for each service.

```text
web
 │
 │
 ▼
db
```

No IP addresses required.

---

# Persistent Database

You'll also want the database to survive container removal.

Add a volume:

```yaml
services:
  db:
    image: postgres:17
    environment:
      POSTGRES_DB: mydb
      POSTGRES_USER: myuser
      POSTGRES_PASSWORD: mypassword

    volumes:
      - postgres_data:/var/lib/postgresql/data

volumes:
  postgres_data:
```

Now:

```text
Container Deleted

↓

Database Still Exists
```

# Typical Django Project Structure

A Dockerized Django project often looks like this:

```text
project/
│
├── Dockerfile
├── docker-compose.yml
├── .dockerignore
├── requirements.txt
├── manage.py
├── core/
├── myApp/
└── static/
```

This is a very common layout you'll see in professional projects.

# Your Practice

1. Create `docker-compose.yml` with:

```yaml
services:
  web:
    build: .
    ports:
      - "8000:8000"
```

2. Run:

```bash
docker compose up
```

3. Stop it with:

```bash
docker compose down
```

4. Then run it in the background:

```bash
docker compose up -d
```

5. Check the status:

```bash
docker compose ps
```
