# Production Dockerfile + Gunicorn

## Why not `runserver`?

So far we've been using:

```bash
python manage.py runserver
```

Django itself tells us:

> "This is a development server. Do not use it in a production setting."

Why?

Because `runserver` is designed for:

* debugging
* auto-reloading
* development

It is **not** optimized for serving many users.

---

# What is Gunicorn?

Gunicorn (Green Unicorn) is a production-grade WSGI server.

Instead of:

```text
Browser
    │
    ▼
Django runserver
```

we'll use:

```text
Browser
    │
    ▼
Gunicorn
    │
    ▼
Django
```

In production, the stack is usually:

```text
Internet
    │
    ▼
Nginx
    │
    ▼
Gunicorn
    │
    ▼
Django
```

We'll add Nginx in the next step.

# Install Gunicorn

Inside your project:

```bash
pip install gunicorn
```

Update your requirements:

```bash
pip freeze > requirements.txt
```

---

# Update the Dockerfile

Your development Dockerfile probably looks like this:

```dockerfile
FROM python:3.12-slim

WORKDIR /app

COPY requirements.txt .

RUN pip install --no-cache-dir -r requirements.txt

COPY . .

EXPOSE 8000

CMD ["python", "manage.py", "runserver", "0.0.0.0:8000"]
```

Replace the last line with:

```dockerfile
CMD ["gunicorn", "core.wsgi:application", "--bind", "0.0.0.0:8000"]
```

> Replace `core` with your Django project package if it has a different name. In your project, `core` appears to be correct.

---

# What does this command mean?

```bash
gunicorn core.wsgi:application --bind 0.0.0.0:8000
```

Breakdown:

### `gunicorn`

Start the Gunicorn server.

---

### `core.wsgi:application`

Gunicorn needs Django's WSGI application object.

Inside your project:

```text
core/
    wsgi.py
```

You'll find:

```python
application = get_wsgi_application()
```

Gunicorn imports that object to serve your application.

---

### `--bind`

Tell Gunicorn which address and port to listen on.

```text
0.0.0.0
```

means:

> Accept connections from outside the container.

---

Because you changed the Dockerfile and installed Gunicorn:

```bash
docker compose up --build
```

or

```bash
docker build -t mysite .
```

# Run Multiple Workers

One of Gunicorn's biggest advantages is that it can run multiple worker processes.

Example:

```bash
gunicorn core.wsgi:application \
    --bind 0.0.0.0:8000 \
    --workers 3
```

Now three workers can handle requests concurrently.

A common starting point is:

```text
workers = (2 × CPU cores) + 1
```

You can tune this later based on your server's resources and workload.

---

# Environment Variables in the Command

Instead of hardcoding values, you can use Compose:

```yaml
command: >
  gunicorn
  core.wsgi:application
  --bind 0.0.0.0:8000
  --workers=3
```

This keeps your `Dockerfile` simpler and allows different commands for development and production.

# Development vs Production

## Development

```text
VS Code

↓

runserver

↓

Auto reload
```

Great for coding because changes appear immediately.

---

## Production

```text
Gunicorn

↓

Fast

↓

Stable

↓

Multiple workers
```

No auto-reload, but much better suited for serving users.

---

# Image Size

A common production improvement is to avoid unnecessary files in the image.

For example:

```dockerfile
COPY requirements.txt .

RUN pip install -r requirements.txt

COPY . .
```

This ordering helps Docker cache dependency installation, so rebuilding after code changes is much faster.

# Verify Gunicorn

Run:

```bash
docker compose up --build
```

You should see output similar to:

```text
[INFO] Starting gunicorn
[INFO] Listening at: http://0.0.0.0:8000
[INFO] Booting worker
```

If you see that, Gunicorn is serving your Django application.
