# Step 2: What happens during `docker build`?

When you run:

```bash
docker build -t mysite .
```

Docker performs these steps:

```
Read Dockerfile
        │
        ▼
Download Python image (if needed)
        │
        ▼
Create Layer 1
        │
        ▼
Create /app directory
        │
        ▼
Create Layer 2
        │
        ▼
Copy requirements.txt
        │
        ▼
Create Layer 3
        │
        ▼
Install packages
        │
        ▼
Create Layer 4
        │
        ▼
Copy Django project
        │
        ▼
Create Layer 5
        │
        ▼
Save final image
```

Notice that Docker builds your image **one instruction at a time**.

# What are Docker Layers?

Every important instruction creates a **layer**.

For example:

```dockerfile
FROM python:3.12-slim
```

↓

Layer 1

---

```dockerfile
WORKDIR /app
```

↓

Layer 2

---

```dockerfile
COPY requirements.txt .
```

↓

Layer 3

---

```dockerfile
RUN pip install -r requirements.txt
```

↓

Layer 4

---

```dockerfile
COPY . .
```

↓

Layer 5

---

You can imagine them like transparent sheets stacked on top of each other:

```
-------------------------
Your Django Code
-------------------------
Installed Packages
-------------------------
requirements.txt
-------------------------
/app Folder
-------------------------
Python Image
-------------------------
```

The final image is all of these layers combined.

# Why Layers Matter

Suppose you edit only:

```python
views.py
```

and rebuild:

```bash
docker build -t mysite .
```

Does Docker reinstall Django?

**No.**

Docker notices that:

* `requirements.txt` didn't change.
* The `RUN pip install ...` layer is still valid.

So it reuses that layer from its cache.

It only reruns:

```dockerfile
COPY . .
```

This makes builds much faster.

---

## Why do we copy `requirements.txt` first?

This order is intentional:

```dockerfile
COPY requirements.txt .

RUN pip install -r requirements.txt

COPY . .
```

Imagine you change only:

```python
models.py
```

If you had written:

```dockerfile
COPY . .

RUN pip install -r requirements.txt
```

then **every code change** would force Docker to reinstall all Python packages.

By copying `requirements.txt` first, Docker can reuse the installed dependencies as long as that file hasn't changed.

This is one of the most common Docker optimizations.

# What's the difference between an Image and a Container?

Imagine you write:

```python
class Student:
    pass
```

This class definition is like a Docker **image**. It's a template.

Now create an object:

```python
student1 = Student()
```

That's like a Docker **container**—a running instance of the image.

You can even create multiple containers from one image, just as you can create many `Student` objects from one class.

# Check your image

Run:

```bash
docker images
```

You might see something like:

```text
REPOSITORY   TAG      IMAGE ID
mysite       latest   a1b2c3d4...
python       3.12     e5f6g7h8...
```

* `mysite` is the image you built.
* `python:3.12-slim` is the base image it was built from.

# Run your first container

Now start a container from the image:

```bash
docker run -p 8000:8000 mysite
```

Here's what each part means:

* `docker run` → create and start a container.
* `-p 8000:8000` → map port 8000 on your Codespace to port 8000 inside the container.
* `mysite` → the image to use.

When the container starts, Docker executes the `CMD` from your `Dockerfile`:

```dockerfile
CMD ["python", "manage.py", "runserver", "0.0.0.0:8000"]
```

# A few useful commands

After building and running your image, these are worth knowing:

```bash
docker images          # List images
docker ps              # Show running containers
docker ps -a           # Show all containers
docker stop <container_id>
docker rm <container_id>
docker image ls        # Another way to list images
docker image rm mysite # Remove the image
```
