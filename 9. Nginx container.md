# Step 9: Nginx – The Front Door of Your Application

## The Problem

Right now your application looks like this:

```text
Browser
    │
    ▼
Gunicorn
    │
    ▼
Django
```

This works.

So why add Nginx?

Because Gunicorn is **an application server**, not a full-featured web server.

It can serve HTTP requests, but it's not optimized for:

* Serving static files (CSS, JS, images)
* Serving uploaded media
* SSL/TLS termination (HTTPS)
* Compression (Gzip/Brotli)
* Rate limiting
* Reverse proxying
* Load balancing

Nginx is built for these tasks.

# Think of a Restaurant

Imagine a restaurant.

Customers **don't walk into the kitchen**.

Instead:

```text
Customer
    │
    ▼

    │
    ▼

```

Nginx = Waiter

Gunicorn + Django = Chef

The waiter receives requests, decides where they go, and returns the response.

# Reverse Proxy

Nginx is called a **reverse proxy**.

Client:

```text
https://example.com
```

Nginx receives the request.

Instead of replying itself, it forwards it:

```text
localhost:8000
```

to Gunicorn.

Gunicorn processes it.

Nginx sends the response back.

The client never knows Gunicorn exists.

Every CSS request goes through Django.

That's unnecessary.

With Nginx:

```text

↓

```

Django isn't involved.

This is much faster.

# Nginx Configuration

Create:

```text
nginx/
    nginx.conf
```

Example:

```nginx
events {}

http {

    server {

        listen 80;

        location / {

            proxy_pass http://web:8000;

        }

    }

}
```

Let's understand it.

---

## listen

```nginx
listen 80;
```

Nginx listens on port 80.

---

## location /

```nginx
location /
```

Means:

Every request.

---

## proxy_pass

```nginx
proxy_pass http://web:8000;
```

This says:

Forward the request to:

```text
web
```

which is the Django service.

Notice:

Not

```text
localhost
```

Instead:

```text
web
```

because that's the service name.

# Add Nginx to Compose

Eventually your compose file becomes:

```yaml
services:

  nginx:
    image: nginx:latest

    ports:
      - "80:80"

    volumes:
      - ./nginx/nginx.conf:/etc/nginx/nginx.conf

    depends_on:
      - web

  web:
    build: .

  db:
    image: postgres:17
```

Now:

```text
Browser

↓

Port 80

↓

Nginx

↓

Gunicorn

↓

Django
```

# Static Files in Production

Django collects static files into one directory:

```bash
python manage.py collectstatic
```

Suppose they end up in:

```text
/app/staticfiles
```

Nginx serves them directly:

```nginx
location /static/ {

    alias /app/staticfiles/;

}
```

Now:

```text
/static/style.css
```

never reaches Django.

---

# Media Files

Likewise:

```nginx
location /media/ {

    alias /app/media/;

}
```

Uploaded files are served directly by Nginx.

# Why Do We Need Both?

A common question is:

> "If Nginx receives requests, why do we need Gunicorn?"

Because they have different jobs.

| Nginx               | Gunicorn                  |
| ------------------- | ------------------------- |
| Web server          | WSGI application server   |
| Serves static files | Runs Django               |
| Reverse proxy       | Executes Python code      |
| SSL/TLS             | Handles Django requests   |
| Compression         | Returns dynamic responses |

Neither replaces the other—they complement each other.
