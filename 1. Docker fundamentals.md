# Step 1: What is Docker?

Imagine you build a Django project.

Normally, to run it on another computer, that computer needs:

* Python
* pip
* Django
* Pillow
* python-dotenv
* All the packages in `requirements.txt`

Different computers often have different versions, which can cause problems.

Docker solves this by packaging everything together.

Think of it like this:

```
Your Laptop
│
├── Django Project
├── Python
├── Django
├── Pillow
├── sqlite
├── Other packages
└── Settings

↓

Docker Image

↓

Any Computer

↓

Works exactly the same
```

The famous phrase is:

> "It works on my machine."

Docker changes that to:

> "It works everywhere."

# Step 2: Docker's Main Concepts

There are only three things you need to understand initially.

## 1. Image

An **image** is like a blueprint or recipe.

Example:

```
Python 3.12
+
Django
+
Your Code
```

It is **not running**.

Think of it like:

```
Class Blueprint
```

---

## 2. Container

A **container** is a running instance of an image.

Think of:

```
Image
     ↓
docker run
     ↓
Container (Running)
```

Like in Python:

```python
class Car:
    pass

car1 = Car()
```

* `Car` → Image
* `car1` → Container

You can create many containers from one image.

---

## 3. Dockerfile

A Dockerfile is just a list of instructions.

Example:

```
Use Python

↓

Install requirements

↓

Copy my code

↓

Start Django
```

Docker reads this file and builds the image.

---

# Step 3: Create Your First Dockerfile

Create a file named:

```
Dockerfile
```

Paste this:

```dockerfile
FROM python:3.12-slim

WORKDIR /app

COPY requirements.txt .

RUN pip install --no-cache-dir -r requirements.txt

COPY . .

EXPOSE 8000

CMD ["python", "manage.py", "runserver", "0.0.0.0:8000"]
```

Now let's understand every line.

---

## Line 1

```dockerfile
FROM python:3.12-slim
```

This means:

> Start from an existing image that already has Python 3.12 installed.

It's like saying:

```
Give me a fresh computer
+
Python installed
```

---

## Line 2

```dockerfile
WORKDIR /app
```

This creates (if needed) and switches into the `/app` directory inside the container.

It's similar to running:

```bash
mkdir /app
cd /app
```

---

## Line 3

```dockerfile
COPY requirements.txt .
```

This copies **only** your `requirements.txt` file from your project into `/app` inside the image.

At this point, the image contains:

```
/app
    requirements.txt
```

---

## Line 4

```dockerfile
RUN pip install --no-cache-dir -r requirements.txt
```

This executes during the image build.

It's equivalent to:

```bash
pip install -r requirements.txt
```

After this, Django and all your project's dependencies are installed **inside the image**.

---

## Line 5

```dockerfile
COPY . .
```

Now Docker copies the rest of your project:

```
manage.py
core/
myApp/
static/
...
```

into `/app`.

---

## Line 6

```dockerfile
EXPOSE 8000
```

This documents that the application inside the container listens on port `8000`.

It doesn't publish the port by itself—that happens when you run the container with `-p`.

---

## Line 7

```dockerfile
CMD ["python", "manage.py", "runserver", "0.0.0.0:8000"]
```

This is the default command that runs when the container starts.

It's the Docker equivalent of you typing:

```bash
python manage.py runserver 0.0.0.0:8000
```

Using `0.0.0.0` is important because it allows connections from outside the container.

---

# Step 4: Build the Image

Run:

```bash
docker build -t mysite .
```

Notice the `.` at the end.

It tells Docker:

> "Use the current directory as the build context."

Docker will:

1. Read the `Dockerfile`.
2. Pull the Python base image (if needed).
3. Install your dependencies.
4. Copy your project files.
5. Create an image named `mysite`.
