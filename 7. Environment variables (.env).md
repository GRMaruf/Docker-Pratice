# Environment Variables (`.env`)

## Why do we need a `.env` file?

Suppose your `docker-compose.yml` contains:

```yaml
environment:
  POSTGRES_USER: admin
  POSTGRES_PASSWORD: super-secret-password
```

If you push this to GitHub:

```text
GitHub
   │
   ▼
Everyone can see your password 😨
```

That's why we never commit secrets to source control.

---

# What is a `.env` file?

A `.env` file is just a text file containing `KEY=VALUE` pairs.

Create a file named:

```text
.env
```

Example:

```text
DEBUG=True

SECRET_KEY=your-secret-key-here

DB_NAME=mydatabase
DB_USER=myuser
DB_PASSWORD=mypassword
DB_HOST=db
DB_PORT=5432
```

> For learning, these values are fine. For a real deployment, use a long, randomly generated `SECRET_KEY` and a strong database password.

---

# Ignore the `.env` File

Add it to `.gitignore`:

```text
.env
```

Now Git won't upload it.

Verify with:

```bash
git status
```

You shouldn't see `.env` listed as an untracked file.

---

# Update `docker-compose.yml`

Instead of writing every variable individually:

```yaml
environment:
  DB_NAME: mydatabase
  DB_USER: myuser
  DB_PASSWORD: mypassword
```

use:

```yaml
services:
  web:
    build: .
    env_file:
      - .env

  db:
    image: postgres:17
    env_file:
      - .env
```

However, the PostgreSQL image expects variables named:

```text
POSTGRES_DB
POSTGRES_USER
POSTGRES_PASSWORD
```

So a better `.env` for Compose is:

```text
DEBUG=True

SECRET_KEY=replace-with-a-long-random-secret

POSTGRES_DB=mydatabase
POSTGRES_USER=myuser
POSTGRES_PASSWORD=mypassword

DB_HOST=db
DB_PORT=5432
```

Then configure the `db` service like this:

```yaml
db:
  image: postgres:17
  environment:
    POSTGRES_DB: ${POSTGRES_DB}
    POSTGRES_USER: ${POSTGRES_USER}
    POSTGRES_PASSWORD: ${POSTGRES_PASSWORD}
```

And the `web` service:

```yaml
web:
  build: .
  env_file:
    - .env
```

# Read Variables in Django

Install `python-dotenv` if you want Django to load `.env` directly outside Docker:

```bash
pip install python-dotenv
```

For Docker Compose, Django already receives the environment variables from the container, so you can simply use `os.getenv()`.

In `settings.py`:

```python
import os

SECRET_KEY = os.getenv("SECRET_KEY")

DEBUG = os.getenv("DEBUG") == "True"

DATABASES = {
    "default": {
        "ENGINE": "django.db.backends.postgresql",
        "NAME": os.getenv("POSTGRES_DB"),
        "USER": os.getenv("POSTGRES_USER"),
        "PASSWORD": os.getenv("POSTGRES_PASSWORD"),
        "HOST": os.getenv("DB_HOST"),
        "PORT": os.getenv("DB_PORT"),
    }
}
```

---

# Why `os.getenv()`?

Imagine this line:

```python
password = os.getenv("POSTGRES_PASSWORD")
```

It's equivalent to asking the operating system:

> "Do you have an environment variable called `POSTGRES_PASSWORD`?"

If yes, you get the value.

If no, it returns `None` (or a default if you provide one).

mydatabase

# Why This Matters

With environment variables, the same code can run in different environments.

**Development**

```text
DEBUG=True
POSTGRES_DB=dev_db
```

**Production**

```text
DEBUG=False
POSTGRES_DB=production_db
```

No code changes—just different environment values.
